
# Docking Guide: The Dragon's Flagon Mini-Game

This document outlines the technical requirements and strategy for integrating **The Dragon's Flagon** (Three-Dragon Ante & Tavern Games) into the main **Project Chimera (Artificer)** codebase.

## 1. Architectural Alignment

Both projects share a modern React stack, making the "docking" process highly efficient:
- **Framework:** React 18+ (Vite)
- **State Management:** Zustand (Slice-based architecture)
- **Styling:** Tailwind CSS
- **Assets:** WebP Sprite Atlases

## 2. Component Integration

The mini-game is encapsulated and can be mounted as a sub-module.

### Mounting the Tabletop
The `TableTop.tsx` component is the main entry point for gameplay. In Artificer, this should be triggered by an interaction event (e.g., "Sit at Table" or "Challenge NPC").

```tsx
// Example Artificer Integration
{activeMode === 'minigame_tda' && (
  <div className="minigame-overlay">
    <TableTop onExit={handleReturnToOverworld} />
  </div>
)}
```

## 3. State & Logic Sync

The mini-game uses `useGameStore.ts`. When docking, certain slices of state should be synchronized with the Artificer main store.

### A. NPC Injection
Instead of using hardcoded patrons, the main game should pass the `NPCData` of the character the player is interacting with.
- **Source:** Artificer `npcSlice`
- **Sink:** TDA `useGameStore.setNPC(npcId)`

### B. Skill Mapping
Artificer character stats should influence the mini-game's special actions:
- **Bluff (CHR):** Reduces gold pay-in during stakes.
- **Sleight of Hand (DEX):** Increases gold stolen from the pot.
- **Concentration (INT):** Reduces ante requirements.

### C. Currency Persistence
The "Gold" in the mini-game must be tied to the player's actual inventory in Artificer.
- **Conversion:** 1 TDA Gold = 1 GP (Gold Piece).
- **Session End:** On exit, the delta (winnings/losses) must be applied to the Artificer `inventorySlice.playerWealth`.

## 4. Artificer Forge & NPC Vault

The **Artificer Forge** (`ArtificerForge.tsx`) is already wired to communicate with the `gemini-2.0-flash` proxy and the GitHub Vault.

- **Generation:** NPCs generated in the Forge use the `GeminiService.generateNPC` method.
- **Syncing:** The `pushToVault` method ensures that new TDA-ready NPCs are committed to the `metadata.json` of the main Artificer repository.
- **Usage:** Once pushed, these NPCs become available in the `NPCShowcase` and can be utilized as opponents with unique traits generated by the AI.

## 5. Visual & Audio Continuity

### Sprite Atlases
The TDA project uses a 5x5 Sprite Atlas for cards. Artificer uses 10x10 atlases for items.
- **Compatibility:** Keep the TDA atlas (`tda_cards.webp`) separate to avoid ID collisions with main game items.
- **Styling:** Ensure the `Cinzel` and `UnifrakturCook` fonts are imported in Artificer's `index.html`.

### Audio Mixing
The `AudioManager` and `Mixer` components should be unified.
- TDA uses a `soundService.ts` proxy. This should be redirected to call Artificer's `audioSlice.playSound` to ensure volume settings and master buses are respected.

## 6. Checklist for Artificer Readiness

- [x] **Secure API Proxy:** Gemini calls are moved to the backend to protect keys.
- [x] **Modular Store:** Game logic is separated from UI.
- [x] **NPC Sync:** Forge successfully pushes NPCs to the Artificer vault.
- [x] **Responsive Tabletop:** UI scales to the Artificer 16:9 tactical container.
- [x] **Solitaire Fallback:** Memory and Solitaire games are available when no NPCs are nearby.
